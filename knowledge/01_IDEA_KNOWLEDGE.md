# IDEA Knowledge

This file is the craft reference for the MAMILAS IDEA agent. It is not a behavior
rulebook. Use it to improve creative decision quality.

## Core Job

The IDEA agent is not a "choose a style" agent. Style and world are usually
already selected by the site. IDEA answers this question:

What idea architecture best explains this source inside the selected path/world?

A strong IDEA output clarifies three things:

- metaphor: what does the source claim become on screen?
- proof mechanism: how does the viewer see that it is true?
- emotional promise: what feeling does the scene leave behind?

## Reading Site Language

If `DIRECTOR MANDATE` exists, it is the Phase 0 decision record. Do not replace
it; sharpen it.

If `SCENE DOSSIER` exists, the site has already produced the raw scene skeleton.
Usually do not rewrite it from scratch. You may move it into a clearer
architecture that explains the source better.

`RENDER LOCK` is not something the IDEA agent changes. IDEA only explains how the
route and scene architecture serve that render law.

## What Route Means

Route is not "more cinematic", "more premium", or "more colorful". Route is a
director decision:

- product film: product proof, lifestyle use, social proof, quiet luxury,
  problem-solution
- education: open the mechanism, inside the object, calm lesson, character-led
  discovery
- documentary: observer, portrait, place-first, testimony
- stylized: mythic drama, graphic proof, game-cinematic momentum

If three routes are truly distinct, the same source produces three different
shooting ideas.

## Scene / Format Architecture

For video, each scene line needs:

- source meaning
- dominant subject
- visible event
- proof/result
- camera or vantage
- lock state

For design, each format needs:

- dominant message/visual
- reading order
- copy placement
- image crop
- negative-space job
- distance/thumbnail behavior

## Creativity Control

A good IDEA agent preserves locks without becoming timid. If the source says
"teach children the water cycle", "a water drop moves" is not enough. Evaporation,
condensation, and collection need visible proof mechanics. If the source says
"the product creates trust", the product cannot merely look beautiful; hands,
surface behavior, use, or result must prove trust.

## Failure Patterns

- three routes that only change names
- treating Render World as route
- forgetting the source claim and writing atmosphere
- deleting Director Mandate
- treating Reference DNA as source
- writing an ad slogan instead of scene architecture

## Good Repair

Repair is not abstract critique. It gives directly usable architecture:

`PROBLEM:` Route reads like a style variant.  
`REPLACE WITH:` Route B = "Problem-solution proof: the previous friction is
visible first, the product behavior becomes the single physical proof, and the
result settles quietly in the final hold."  
`VERIFY:` Routes A/B/C carry distinct proof mechanisms.
