# MAMILAS — Three-Phase Completion Map

Date: 2026-07-15

This is the navigation document for the next three focused Codex goals. It does not declare any
phase complete. Each new chat audits only its active phase, closes it with evidence, writes a receipt,
then stops. Existing dirty-worktree changes are preserved; no reset, checkout, stash, push or blind
rewrite is allowed.

## Immutable product result

- Mami's raw source and explicit choices survive losslessly.
- The site records decisions and evidence; it does not infer intent from source words and does not
  write final engine prompts.
- Claude/Codex writes final prompts only inside the validated command lifecycle.
- Forty-six worlds remain real physics packets; compatible refs and palette-as-light reach the author.
- No external generation API, automatic generation, batch, upscale pipeline or agent loop.
- Motion opens only for a current real frame carrying Mami `APPROVE`.
- Green tests prove implementation, not visual quality. A real Mami frame verdict remains sovereign.

## Sol operating charter

Mami owns the desired product feeling and outcome. Sol owns the architecture and the route used to
reach it. Names, fields, task lists, earlier recommendations and even this map's suggested mechanisms
are not sacred. When Sol finds a stronger idea, it should develop, test and implement it while
preserving the immutable product result.

Sol is allowed to make reversible experiments, discover that an idea was wrong and correct it from
evidence. This is not permission to hide failures or overwrite unrelated work. It is permission to
avoid paralysis: a phase does not wait for an imaginary 10/10 score. It closes when its user outcome
works, proportionate gates pass and no known critical regression remains. Secondary polish findings
go to the final convergence ledger.

Sol may organize bounded internal agents however the active work benefits: parallel research,
implementation, testing or comparison. There is no fixed agent topology and Mami does not need to
approve or watch internal coordination. Self-perpetuating agent loops and empty usage-burning debate
remain forbidden.

After each builder phase, one fresh independent auditor agent reviews only that phase and writes a
report. The auditor does not begin an endless repair loop. Data loss, security, forbidden product
behavior or a new broken gate is repaired immediately; other actionable findings accumulate for the
single Final Convergence & Delivery session after all three phases.

## Closure order

The product has three domains. Close them in dependency order: **Core → Studio → Command**. Each phase
produces a builder receipt and one independent audit report. Then a fresh fourth session—an integration
session, not a new architecture phase—fixes the combined audit ledger and delivers the site.

### Phase 1 — Decision Core & Creative Library

Owns:

- `src/core/source.ts`, `pure.ts`, `brain.ts`, `contract.ts`, `engine.ts`
- `SURGERY_DATA.json`, paths, projects, presets, 46 worlds, refs, palettes and music data
- lossless source/beat identity, recipe defaults, WorldPacket, ref compatibility, palette-as-light
- deterministic storyboard/brief data and prompt-boundary inputs

Does not own:

- page layout, visual polish, navigation or browser interaction
- provider launchers, agent adapters or artifact sessions

Finish evidence:

- all 46 worlds produce unique valid WorldPackets; legacy render law and compatible refs survive
- no source-intent regex/NLP/word-count inference in the production path
- selected world/ref/palette data reaches canonical output without hex/IP/drift
- representative real `generateBatch` outputs are inspected, not only unit tests
- TypeScript, focused tests and full Vitest pass; `PHASE-1-CORE.md` receipt and
  `PHASE-1-AUDIT.md` independent report are written

### Phase 2 — Studio Application, UX & Evidence State

Owns:

- React pages/components/styles and the Zustand store
- Brief → Director → Recipe → Scenes → Timeline → QA navigation
- canonical readiness, storyboard approval, agent-prompt import, frame byte hash/dimensions/verdict
- stale invalidation, Project Pack round-trip, closeout, accessibility and responsive behavior
- browser-visible labels, disabled states, errors and export placement

Does not own:

- rewriting world/ref/palette laws already closed in Phase 1
- agent role prose, provider behavior or command artifact orchestration

Finish evidence:

- a real browser flow proves source → choices → scenes → prompt import → shot approval → frame upload
  → SHA-256/dimensions → Mami `APPROVE` → motion open → decision change/stale close
- Project Pack export/import restores decisions, scenes, prompt/frame receipts and the same gate result
- no sidebar/export/readiness bypass; Windows/mobile-visible UX findings are either fixed or explicit
- TypeScript, full Vitest, build and full Playwright pass; `PHASE-2-STUDIO.md` receipt and
  `PHASE-2-AUDIT.md` independent report are written

### Phase 3 — Command & Manual Production Runtime

Owns:

- `commandExport.ts`, `agentProtocol.ts`, `agents/PROTOCOL.md`
- `scripts/mamilas-command.mjs`, runner mirrors, Windows/macOS launchers
- Claude/Codex adapters, role cards, workspace artifacts and the `mamilas-uret` skill
- exact `MamiDirectives`, minimum role context, author→jury lifecycle and post-session validation

Does not own:

- site visual redesign or creative-library rewrites
- image/video generation, provider APIs, automatic loops or paid-credit rituals

Finish evidence:

- Timeline and QA export the same `mamilas.command.v2026`; legacy production JSON is non-runnable
- protocol/decision/storyboard/content/input hashes reject tamper and stale evidence
- each scene runs only its next valid role; jury vocabulary is three-valued; one revision maximum
- image author never receives the site's scaffold prompt; motion cannot open before current real frame
- Claude/Codex evidence parity, dry-run selection, interactive stub launch and post-session artifact
  validation pass; runner mirrors are byte-identical
- TypeScript, full Vitest, build and relevant E2E pass; `PHASE-3-COMMAND.md` receipt and
  `PHASE-3-AUDIT.md` independent report are written

## New-chat operating rule

Every new chat reads, in order:

1. `AGENTS.md`
2. `docs/ai/PROJECT_CONTRACT.md`
3. `artifacts/decision-pipeline-implementation/EXECUTION_STATE.md`
4. this file
5. only the active phase's code and last receipt

It first performs a read-only audit of that phase, then Sol is free to improve the solution rather than
blindly preserve its proposed mechanism. It must not reopen a completed phase unless it has a concrete
cross-phase regression with file/output evidence. Auto-compaction is harmless because the state file,
phase receipt and audit report—not chat memory—define the next action.
